# Name-prefix conventions (CRITICAL — read carefully)

The KashFlow team uses two name prefixes as control signals. Respect them exactly.

- **Single asterisk `*` (e.g. `*Acme Cards`)** = customer has ceased trading. Sync the record but force `hidden=true, status='inactive'`. Keep the `*` prefix in `company_name` so it's still identifiable if someone shows hidden customers.
- **Double asterisk `**` (e.g. `**Acme Cards`)** = active territorial protection. Sync normally. Set `territorial_protection=true`. **KEEP the `**` prefix in `company_name`** — this is how the user identifies them at a glance when sorting alphabetically. Do NOT strip it.
- **No prefix** = normal customer. Sync normally. `territorial_protection=false`.

**WARNING: Dave's `five_minute_customers` pipeline strips prefixes!** When copying from KashFlow, it converts `**` to `*` and removes regional suffixes like `(LD)`. This means `five_minute_customers.Name` is NOT reliable for prefix detection. **Always use the raw `brainbox_candy.customers.Name` field** (joined on CustomerID) for prefix detection and for writing to `company_name` in Supabase.

Detection rules in SQL (MariaDB), using the authoritative raw `customers.Name`:
- `single_asterisk = c.Name LIKE '*%' AND c.Name NOT LIKE '**%'`
- `double_asterisk = c.Name LIKE '**%'`

# Steps

## Step 1 — Determine sync mode

Read Supabase `sync_state`:
- key=`kashflow_customer_last_sync`
- key=`kashflow_customer_last_full_sweep`

- If `last_sync IS NULL` -> **full sweep** (first run ever)
- If today is Sunday AND `last_full_sweep < today's date` -> **full sweep** (weekly refresh)
- Otherwise -> **delta**

## Step 2 — Pull MySQL data

- **Delta mode**: `WHERE fmc.last_updated >= '<last_sync - 1 day>'`
- **Full sweep mode**: no WHERE clause (all ~7,077 rows)

**IMPORTANT: Dave's `five_minute_customers` strips the `**` and `*` prefixes from `Name` (converts `**` to `*`, removes regional suffixes like `(LD)`, etc.). For territorial-protection detection and the canonical `company_name`, we MUST use the Name from the raw `customers` table. Use `five_minute_customers` for EVERYTHING ELSE (geocoding, addresses, phone, etc.).**

Join to `customer_sources` for source name AND to raw `customers` for the authoritative Name:
```sql
SELECT
  fmc.CustomerID,
  fmc.Code,
  COALESCE(c.Name, fmc.Name) AS Name,   -- authoritative name (with ** and * prefixes preserved)
  fmc.Telephone, fmc.Mobile, fmc.Email, fmc.Website,
  fmc.Street_Address, fmc.Street_Address_2_Town, fmc.Town, fmc.Town_County,
  fmc.CountryName, fmc.Postcode,
  fmc.Notes, fmc.Discount, fmc.PaymentTerms, fmc.Created,
  fmc.VATNumber, fmc.EORI, fmc.Member_No, fmc.PO_Number,
  fmc.DeliveryAddress1, fmc.DeliveryAddress2, fmc.DeliveryAddress3, fmc.DeliveryAddress4,
  fmc.DeliveryPostcode, fmc.Store_Type,
  fmc.lat, fmc.lng, fmc.formatted_address,
  fmc.last_updated,
  cs.Name AS source_name
FROM five_minute_customers fmc
LEFT JOIN customers c ON c.CustomerID = fmc.CustomerID
LEFT JOIN customer_sources cs ON cs.ID = fmc.Source
WHERE <delta clause as above>
ORDER BY fmc.Code;
```

For full sweeps, page through in batches of ~200 rows to avoid timeouts.

## Step 4 — Compare and apply

For each MySQL row:

**Determine prefix flags:**
- `single_asterisk = Name starts with '*' but not '**'`
- `double_asterisk = Name starts with '**'`

**If no matching Supabase row** (new insert):
- Build full payload with all synced columns
- `company_name = Name` (preserve prefix)
- `territorial_protection = double_asterisk`
- If `single_asterisk`: also set `hidden=true, status='inactive'`
- Else: `hidden=false, status='prospect'`
- Write `kf_synced_at = now(), created_at = now(), updated_at = now()`

**If matching row exists** (update):
- Compare each KashFlow-owned column (see mapping table)
- Always compare and potentially update `territorial_protection` (derived fresh from Name)
- For `hidden` and `status`: only write them if `single_asterisk` is true (force to true/'inactive'). Otherwise leave whatever they were.
- For `customer_notes`: skip entirely on updates (CRM may have edited).
- If any column differs, build UPDATE payload with ONLY changed columns + `updated_at = now()`.
- If no differences -> skip (counts as unchanged).

## Step 6.5 — Refresh proximity conflicts (5 Min Walk cache)

After the main sync is complete and `sync_state` is updated, refresh the cached proximity data in `customers.conflict_kf_code` and `customers.conflict_info`. This keeps the CRM's "5 Min Conflict" column in sync with the latest lat/lng data.

Run these three SQL statements against Supabase in order. The full refresh typically completes in under 5 seconds.

**Step 6.5a — Clear existing conflict cache:**

```sql
UPDATE customers 
SET conflict_kf_code = NULL, 
    conflict_info = NULL 
WHERE conflict_kf_code IS NOT NULL;
```

**Step 6.5b — Populate threat flag for territorial (`**`) customers:**

```sql
WITH territorial AS (
  SELECT id, lat, lng
  FROM customers
  WHERE territorial_protection = true AND lat IS NOT NULL
),
candidates AS (
  SELECT id, kf_code, company_name, lat, lng, status, last_order_date
  FROM customers
  WHERE territorial_protection = false AND hidden = false AND lat IS NOT NULL
),
pairs AS (
  SELECT 
    t.id AS terr_id,
    c.kf_code AS nearby_code,
    c.company_name AS nearby_name,
    c.last_order_date,
    c.status,
    ROUND(111320 * SQRT(
      POWER(t.lat - c.lat, 2) + 
      POWER((t.lng - c.lng) * COS(RADIANS((t.lat + c.lat) / 2)), 2)
    ))::int AS dist_m,
    ROW_NUMBER() OVER (
      PARTITION BY t.id 
      ORDER BY (
        CASE c.status WHEN 'active' THEN 0 WHEN 'dormant' THEN 1 ELSE 2 END,
        POWER(t.lat - c.lat, 2) + POWER((t.lng - c.lng) * COS(RADIANS((t.lat + c.lat) / 2)), 2)
      )
    ) AS rn
  FROM territorial t CROSS JOIN candidates c
  WHERE c.lat BETWEEN t.lat - 0.005 AND t.lat + 0.005
    AND c.lng BETWEEN t.lng - 0.008 AND t.lng + 0.008
)
UPDATE customers cu
SET conflict_kf_code = p.nearby_code,
    conflict_info = jsonb_build_object(
      'company_name', p.nearby_name,
      'last_order_date', p.last_order_date,
      'status', p.status,
      'distance_m', p.dist_m,
      'type', 'threat'
    )
FROM pairs p
WHERE cu.id = p.terr_id AND p.rn = 1 AND p.dist_m <= 417;
```

**Step 6.5c — Populate zone flag for non-territorial customers:**

```sql
WITH territorial AS (
  SELECT id, kf_code, company_name, lat, lng, last_order_date
  FROM customers
  WHERE territorial_protection = true AND lat IS NOT NULL
),
candidates AS (
  SELECT id, lat, lng
  FROM customers
  WHERE territorial_protection = false AND hidden = false AND lat IS NOT NULL
),
pairs AS (
  SELECT 
    c.id AS cand_id,
    t.kf_code AS terr_code,
    t.company_name AS terr_name,
    t.last_order_date,
    ROUND(111320 * SQRT(
      POWER(t.lat - c.lat, 2) + 
      POWER((t.lng - c.lng) * COS(RADIANS((t.lat + c.lat) / 2)), 2)
    ))::int AS dist_m,
    ROW_NUMBER() OVER (
      PARTITION BY c.id 
      ORDER BY POWER(t.lat - c.lat, 2) + POWER((t.lng - c.lng) * COS(RADIANS((t.lat + c.lat) / 2)), 2)
    ) AS rn
  FROM territorial t CROSS JOIN candidates c
  WHERE c.lat BETWEEN t.lat - 0.005 AND t.lat + 0.005
    AND c.lng BETWEEN t.lng - 0.008 AND t.lng + 0.008
)
UPDATE customers cu
SET conflict_kf_code = p.terr_code,
    conflict_info = jsonb_build_object(
      'company_name', p.terr_name,
      'last_order_date', p.last_order_date,
      'distance_m', p.dist_m,
      'type', 'zone'
    )
FROM pairs p
WHERE cu.id = p.cand_id AND p.rn = 1 AND p.dist_m <= 417;
```

# Error handling

- If MySQL is unreachable: log to `customer_sync_log` with `error_count=0` and `errors=[{top_level:"MySQL unreachable"}]`, do NOT update `sync_state`. Return the error clearly.
- If any individual row fails to write: record in `errors` array, continue with the rest. Only treat as total failure if >10% of rows error.
- Never retry destructive operations. Log and move on.
